# ОТЧЁТ

## Этап №4 | AI/ML | Создание прототипа НС и получение первой точности распознавания

**Проект:** TeachAI — интеллектуальная система персонализированного обучения  

---

## 1. Тема и описание задачи

**Тема:** создание прототипа AI-системы на основе предобученной языковой модели (LLM) для автоматической генерации и контроля качества образовательного контента.

**Описание задачи.**

TeachAI — прикладная **inference-система**: она не обучает собственную нейросеть с нуля и не выполняет fine-tuning. Используется готовая GPT-модель (OpenAI API) для:

- генерации плана курса, уроков, примеров кода, тестов;
- ответов на вопросы студента в рамках урока;
- проверки релевантности вопросов теме урока;
- валидации и автокоррекции сгенерированного контента.

**Связь с формулировкой этапа №4.**

Классический сценарий этапа (обучение собственной CNN/классификатора на размеченной выборке, кривые loss/accuracy по эпохам) к данному проекту **не применим** — это следует из архитектурных решений этапов №2–№3 (см. приложения в корне репозитория: `Об отсутствии сбора базы и дообучения.md`, `Об отсутствии парсинга.md`).

Результат этапа №4 для TeachAI:

- **прототип**, демонстрирующий полный цикл работы системы;
- использование **предобученной** нейросети (GPT) через API;
- механизмы **контроля качества** на этапе inference (валидация, перегенерация), а не обучение весов модели.

**Важно:** формальное исследование точности (hold-out выборка, precision/recall, кривые обучения) на момент сдачи отчёта **не проводилось**. Количественные проценты качества в проектной документации ранее присутствовали ошибочно и в данном отчёте **не используются**.

## 2. База данных

### 2.1. Обучающая база для НС — отсутствует (обоснованно)

По архитектурному решению проекта:

- **не собирается** размеченная обучающая выборка;
- **не выполняется** дообучение GPT;
- **не используется** парсинг внешних источников для формирования датасета.

Причины:

1. Задачи решаются промпт-инженерией, валидацией и перегенерацией без fine-tuning.
2. Масштаб дипломного проекта недостаточен для значимого улучшения модели методом fine-tuning.
3. Этические и организационные ограничения на сбор пользовательских диалогов без согласия.
4. Экономика: fine-tuning GPT требует отдельного формата данных и существенных затрат.

**Вывод:** раздел «База» в классическом ML-смысле (изображения, разметка, train/val/test) **не применим**. Конфиденциальных обучающих данных нет, потому что обучающая база **не создаётся по замыслу системы**.

### 2.2. Операционные данные (не для обучения НС)

| Источник | Назначение |
|----------|------------|
| Профиль и прогресс (`state_manager.py`, JSON) | Персонализация, сохранение уроков и оценок |
| Промпты в генераторах (`lesson_generator.py`, `course_plan_generator.py` и др.) | Параметризация запросов к API |
| Логи (`logs/teachai.log`) | Отладка, учёт ошибок API и валидации |
| `.env` | API-ключ, модель (`LLM_MODEL`), прокси |

Эти данные **не используются** для обучения или дообучения нейросети.

## 3. Параметризация данных

В TeachAI «параметризация» — это **настройка входа inference**, а не аугментация изображений или нормализация тензоров для обучения.

### 3.1. Параметры пользователя и курса

- имя, общее время обучения, длительность урока;
- стиль общения: `formal` | `friendly` | `casual` | `brief`;
- выбранный курс, текущий урок, история тестов.

### 3.2. Параметры запросов к LLM

| Параметр | Типичное значение | Назначение |
|----------|-------------------|------------|
| `LLM_MODEL` | `gpt-4o-mini` (по умолчанию) | Базовая модель |
| `temperature` | 0.7 — генерация контента; 0.2 — проверка релевантности | Креативность vs. стабильность |
| `max_tokens` | до 3500 | Ограничение длины ответа |
| `response_format` | `{"type": "json_object"}` | Структурированный вывод (планы, релевантность) |

Реализация: `content_utils.py`, метод `make_api_request`.

### 3.3. Структурная параметризация контента

- JSON-схемы плана курса (разделы → темы → уроки);
- список примеров `{"title", "description", "code"}` с правилами в `examples_validation.py` (минимум 3 примера, не менее 3 строк кода, запрет JS/HTML-маркеров);
- проверка Python через `ast` (синтаксис без выполнения).

### 3.4. Параметры контроля качества

- валидация и автокоррекция плана (`course_plan_generator.py` → `_validate_and_fix_course_plan`);
- эвристики релевантности (`relevance_checker.py`);
- перегенерация при провале валидации (`examples_validation.py` → `validate_and_regenerate_if_needed`).

## 4. Архитектура нейросети

### 4.1. Используемая НС (внешняя, предобученная)

**Модель:** GPT (семейство decoder-only Transformer), доступ через OpenAI Chat Completions API.

Упрощённая схема (концептуально):

```
Вход: токенизированный промпт (system + user messages)
  → Embedding
  → N × (Multi-Head Self-Attention + Feed-Forward + Layer Norm)
  → LM Head
  → вероятностное предсказание следующего токена
  → текст / JSON на выходе
```

Веса модели **не обучаются в проекте**; обучение выполнено производителем (pre-training и дальнейшая донастройка на стороне OpenAI).

### 4.2. Архитектура программного прототипа TeachAI

```
TeachAIEngine (engine.py)
    ├── ConfigManager, StateManager
    ├── ContentGenerator (фасад)
    │     ├── CoursePlanGenerator
    │     ├── LessonGenerator
    │     ├── ExamplesGenerator + ExamplesValidation
    │     ├── AssessmentGenerator
    │     ├── QAGenerator
    │     └── RelevanceChecker
    ├── Assessment (проверка знаний)
    └── UI (ipywidgets, Jupyter)
```

**Точка входа:** `TeachAI.ipynb` (оркестратор — `engine.py`, класс `TeachAIEngine`).

Прототип — это **оркестрация inference** и **постобработка**, а не отдельно обученный классификатор на собственной выборке.

## 5. Графическое подтверждение

Классических кривых `loss` / `accuracy` по эпохам обучения **нет** — локальное обучение нейросети не выполнялось.

### 5.1. Что подтверждает выполнение этапа

1. **Схема архитектуры** — поток: промпт → API → валидация → отображение в UI (можно вставить рисунок из draw.io / PowerPoint).
2. **Скриншоты работающего прототипа** (рекомендуется приложить к Word-версии отчёта):
   - настройка профиля;
   - сгенерированный план курса;
   - фрагмент урока с примерами кода;
   - контрольное задание / результат теста;
   - проверка релевантности вопроса (модуль `relevance_checker.py`).
3. **Фрагмент лога** `logs/teachai.log` — успешная генерация, срабатывание валидатора или перегенерация (при наличии).
4. **Демонстрация валидации** — пример, когда контент не прошёл проверку и система запросила перегенерацию (если воспроизводится на демо-сценарии).

### 5.2. Количественные метрики

**Не приводятся:** исследование точности генерации (проценты успешных планов, кода, примеров) **не проводилось**. Любые ранее упомянутые в черновиках документации проценты не являются результатами эксперимента и в отчёт не включены.

Формулировка для сдачи:

> Подтверждение этапа №4 — **работоспособный прототип** и демонстрация цепочки «запрос → генерация → валидация → интерфейс», а не численная accuracy обученной с нуля модели.

## 7. Выводы

1. **Прототип создан и работает** — TeachAI демонстрирует цикл: настройка профиля → план курса → урок → примеры → тесты → ответы на вопросы с проверкой релевантности.

2. **Собственная нейросеть не обучалась** — осознанное архитектурное решение; используется предобученная GPT с промпт-инженерией и многоуровневой валидацией на этапе inference.

3. **Численная «первая точность распознавания» в классическом ML-смысле не измерялась** — формальный бенчмарк на размеченной выборке не проводился; качество обеспечивается архитектурой (промпты, валидаторы, перегенерация).

4. **Подход соответствует типу проекта** — прикладная LLM-система (inference-based), согласованная с обоснованиями этапов №2–№3.

5. **Ограничения:** зависимость от API и сети; стохастичность ответов LLM.
